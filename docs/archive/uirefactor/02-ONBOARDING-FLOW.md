## Technical Implementation

### Database Changes
```sql
-- Add onboarding_completed flag (if not exists)
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS onboarding_completed BOOLEAN DEFAULT FALSE;
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS username_changes_remaining INTEGER DEFAULT 1;
```

### Onboarding Check (Client-Side Provider)

**NOT middleware.** We use the existing `OnboardingProvider` pattern:

```typescript
// app/[locale]/(main)/_providers/onboarding-provider.tsx
// Check profile.onboarding_completed
// If false â†’ render onboarding modal/flow
// If true â†’ render children normally
```

The onboarding is triggered via:
1. Auth confirm route sets `?onboarding=true` query param
2. OnboardingProvider reads the param or checks `profile.onboarding_completed`
3. Modal/flow renders client-side (not a redirect)

### Route Structure (Modal-Based, Not Separate Routes)

**Why modal, not routes?**
- No URL exposure during onboarding
- User can't bookmark mid-onboarding
- Simpler state management
- Matches existing pattern

## Animation Specs

**Use existing motion patterns only. No new decorative animations.**

- **Screen transitions:** Slide left/right (existing pattern)
- **Card selection:** Scale 1.02 + shadow (existing pattern)
- **Input focus:** Border color transition (existing pattern)
- **Button hover:** Background shift (existing pattern)
- **Completion:** Simple success state (no confetti â€“ forbidden)

**Note:** Rails forbid new decorative animations and gradients.
